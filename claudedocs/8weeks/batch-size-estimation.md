# 스케줄러 배치 크기 산정 근거

## 하류 시스템 처리 한계 분석

| 항목 | 값 | 근거 |
|------|-----|------|
| DB 커넥션 풀 (HikariCP) | 40 (max) / 30 (min-idle) | `jpa.yml` 설정 |
| 주문 1건 평균 처리 시간 | ~200ms | 재고 차감(비관적 락) + 쿠폰 검증 + 주문 저장 + 이벤트 발행 |
| 이론적 최대 TPS | 40 / 0.2 = **200 TPS** | 커넥션 풀 / 평균 처리 시간 |
| 안전 마진 적용 (70%) | 200 × 0.7 = **140 TPS** | 커넥션 고갈 방지, 다른 API 공유 고려 |

> 실제 설정에서는 여유를 두고 **175 TPS**로 설정했습니다.
> Tomcat max threads(200)는 커넥션 풀(40)보다 넉넉하므로 병목은 DB 커넥션입니다.

## Thundering Herd 완화를 위한 분산 발급

1초에 175명을 한 번에 발급하면 175명이 **동시에** `POST /orders`를 호출합니다.

```
AS-IS: 1초마다 175명 동시 발급 → DB 커넥션 175개 동시 요청 → 풀 고갈
TO-BE: 100ms마다 ~18명씩 발급 → 10회에 걸쳐 분산 → 순간 부하 10배 감소
```

| 설정 | 값 | 산정 |
|------|-----|------|
| `queue.scheduler.interval-ms` | 100ms | 1초를 10구간으로 분할 |
| `queue.batch-size` | 18 | 175 TPS / 10 = 17.5 → 올림 18 |
| 실제 초당 발급량 | 18 × 10 = 180명/초 | 안전 TPS(175) 이내 수렴 |

## 토큰 TTL 설정 근거

| 설정 | 값 | 이유 |
|------|-----|------|
| `queue.token-ttl-seconds` | 300초 (5분) | 유저가 주문 폼 작성 + 결제 진행에 충분한 시간 |

- TTL이 너무 짧으면: 유저가 주문 폼 작성 중 토큰 만료 → UX 악화
- TTL이 너무 길면: 미사용 토큰이 슬롯을 점유 → 뒤 순번 유저 대기 시간 증가
- 5분은 주문 작성(~1분) + 결제(~1분) + 여유 마진(3분)

## 최대 대기열 크기

| 설정 | 값 | 이유 |
|------|-----|------|
| `queue.max-queue-size` | 100,000 | 10만명 대기 시 예상 최대 대기 시간 = 100,000 / 175 ≈ 571초 ≈ ~9.5분 |

- 9.5분을 초과하면 유저 이탈률이 급격히 증가하므로 10만명 이상은 거부하는 것이 합리적
- `QUEUE_FULL` 에러로 "잠시 후 다시 시도" 안내

## 예상 대기 시간 계산

```
예상 대기 시간(초) = 내 순번(1-based) / 초당 처리량(175)
```

- 순번 100번: 100 / 175 ≈ 0.57초
- 순번 1,000번: 1,000 / 175 ≈ 5.7초
- 순번 10,000번: 10,000 / 175 ≈ 57초

> 이 수치는 추정값이며, 토큰 미사용(만료) 시 추가 발급이 발생하므로 실제 대기 시간은 더 짧을 수 있습니다.
